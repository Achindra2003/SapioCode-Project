## AI BACKEND — DEEP DIVE (Your Contribution)

### The Orchestrator (`orchestrator.py` — 1047 lines)

The brain of the system. Uses LangGraph's StateGraph — a directed acyclic graph where each node is an async function and edges are conditional.

**Pipeline per request:**
```
Request → AST Sync (tree-sitter parses code) → Router (maps mode to tool) → Tool → END
```

**No LLM is called until the Tool node.** The AST Sync and Router are pure computation — this keeps latency down.

**State management:** LangGraph's `MemorySaver` persists the `messages` list across API calls using `thread_id` as the key. This is how the AI "remembers" what it already told a student — without a database query — so it never repeats the same hints.

**The `_PERSONA` system prompt — 6 absolute rules:**
1. NEVER output code (no backticks, no pseudocode, no function definitions)
2. NEVER give direct answers or confirm/deny correctness
3. Keep responses under 120 words
4. End every response with exactly ONE Socratic question
5. Reference specific line numbers from the student's actual code
6. Detect and ignore system-provided boilerplate (starter code signatures)

**Critic loop (code leak prevention):**
After every LLM response, a regex scans for code markers:
```python
_CODE_MARKERS = re.compile(
    r'```|(?:def |class |function |if\s*\(|for\s*\(|while\s*\()[\w\s]*[({:]',
    re.IGNORECASE
)
```
If code is detected → the prompt is appended with a rewrite instruction and retried (max 2 times). The model cannot leak code to students.

### Affect-Aware Routing

Every request includes `frustration_score` (0.0–1.0) and `editor_context` from the frontend. The backend uses these to select a teaching tone:

| Condition | Tone | What happens |
|-----------|------|-------------|
| frustration ≥ 0.7, idle > 180s, backspace > 40% | `gentle` | Warm, empathetic. Focuses on what student did right. Simple yes/no questions. |
| Default | `socratic` | Probing questions referencing specific line numbers |
| frustration < 0.4, run_count ≥ 5, paste < 3 | `challenge` | Edge cases, time complexity, "why does this work?" |

A student who's been idle for 3 minutes gets a different response than one who's been actively running code — from the same AI, same prompt, same model.

### Progressive Hint Escalation

Controlled by `hint_count` (persisted via MemorySaver):

| Hint # | Level | Behaviour |
|--------|-------|-----------|
| 1 | `conceptual` | High-level approach question. No line references yet. |
| 2–3 | `targeted` | References a specific line. Give a concrete test input to trace through. |
| 4–5 | `scaffolded` | Names the concept needed. Breaks problem into sub-tasks. |
| 6+ | `near_answer` | Plain-English directional nudge. Still no code. |

### Rich System Prompt

Every LLM call receives a structured prompt including: numbered student code, full AST analysis (algorithm pattern, function call graph, issues with Socratic question seeds), failing test cases, compiler output, the teaching focus, affect level, escalation level, and a deduplicated list of concepts already discussed so the AI doesn't repeat itself.

### AST Parser (`ast_parser.py` — 896 lines)

Uses **tree-sitter** bindings for Python, Java, C++, and JavaScript. This is compiled grammar-based parsing — not regex or heuristics.

**What it produces per function:**
- Name, parameters, whether it calls itself (recursion), whether a base case exists (conditional+return inside recursive function), loop count, call graph, return-value presence

**Algorithm pattern detection** (deterministic, from variable names + structure):
- Sees variables named `memo`, `cache`, `dp` + recursion → `dynamic_programming`
- Sees variables `lo/hi`, `left/right` without recursion → `two_pointer`
- Sees `deque`, `queue` import → `BFS_DFS`
- Nested loop depth ≥ 2 → `brute_force`

**Issue locations** — each one includes a Socratic question seed:
```
missing_base_case at line 5 → "When should `fibonacci` stop calling itself?"
no_termination at line 8    → "What condition makes this while loop eventually become False?"
missing_return at line 3    → "What should `solve` give back to the caller?"
```

The LLM receives these seeds in the system prompt. It uses them as inspiration for its question — never copies them verbatim.

### Groq Client (`groq_client.py` — 223 lines)

Dual-model failover with resilience engineering:
- **Primary**: `moonshotai/kimi-k2-instruct` — stronger reasoning
- **Fallback**: `llama-3.3-70b-versatile`
- **Split timeouts**: connect=5s, read=20s — a slow/stalled response doesn't trigger a connection retry
- **Consecutive failure tracking**: after 3 consecutive primary failures, primary is bypassed entirely until it recovers — prevents cumulative 5s+ latency on every request
- **Exponential backoff on fallback**: 1.5s, 3s, 4.5s for rate limit (429) responses
- **Also handles**: Groq Whisper transcription for voice viva answers

### Viva Voce Verification (`viva.py` — 563 lines)

**Question generation is deterministic — no LLM.** Questions are derived directly from the AST:
- Every student-written function gets a purpose or trace question
- Every detected issue gets a line-explanation question
- Every detected algorithm pattern gets a "why did you choose this?" question
- Always adds an edge-case question
- Boilerplate functions (`main`, `__init__`, etc.) are filtered out — only questions about student-written code

**Answer evaluation is LLM-based.** The model receives: the full code, the AST analysis as "ground truth", the question, the student's answer, and expected concepts. It returns a structured JSON verdict (PASS / NEEDS_DETAIL / FAIL) with matched/missing concepts and red flags.

**Fallback**: if the LLM fails during evaluation, keyword-matching against the expected concepts kicks in automatically.

**Verdict thresholds** (configurable):
- PASS: average score ≥ 0.6
- WEAK: average score ≥ 0.4
- FAIL: average score < 0.4

### Frustration Detection (`useTelemetry.ts` + `frustration.ts`)

`useTelemetry` is a React hook that instruments the Monaco editor — listens to every keystroke, paste, and run event, and tracks idle time with a `setInterval`. It returns an `EditorContext` object.

`calculateFrustration` is a pure function:
```
idle > 180s     → +0.40
idle > 60s      → +0.20
backspace > 40% → +0.30
backspace > 20% → +0.15
run_count > 5   → +0.20
paste × 0.10    → max +0.30
                   ──────────
                   clamped to [0.0, 1.0]
```

This score travels with **every** AI request. The backend's `_pick_affect_level()` function reads it alongside the raw `editor_context` signals to select the teaching tone.

## Q&A TALKING POINTS

**"Why Socratic instead of just explaining the answer?"**
> Research by Chi (2001) and VanLehn (2011) on self-explanation effects shows students who generate explanations themselves retain knowledge significantly better than those who receive them. SapioCode forces active reasoning at every step.

**"How do you prevent the AI from giving answers?"**
> Three independent layers: (1) Hardcoded system prompt with 6 absolute rules the model is trained to follow. (2) A regex critic loop that scans every response for code markers. (3) Automatic rewrite-and-retry if code leaks through. All three must fail for a code answer to reach the student.

**"Why tree-sitter instead of just sending code to the LLM?"**
> Tree-sitter gives deterministic, structural ground truth. Before the LLM sees anything, we already know: which function is recursive, whether a base case exists, which specific line has an infinite loop. This makes hints precise — "look at line 7" — rather than vague. LLMs can hallucinate structural facts about code; tree-sitter cannot.

**"How does viva stop plagiarism?"**
> Questions are generated from the student's specific code — their variable names, their function structure, their detected issues. Two students who submit identical code will face different questions because the questions reference specific identifiers and lines. The LLM evaluates semantic intent, not keyword matching — a student who memorised "it uses recursion" will fail if they can't explain why the base case is needed.

**"What if the AI service goes down mid-exam?"**
> Dual-model failover (Kimi K2 → Llama 3.3) means a single model outage is invisible. Consecutive failure tracking means we don't keep hammering a dead model. The frontend has a 45-second timeout, 2 automatic retries, and a visible "↻ Retry" button. Students never see a blank failure — they see a helpful error with a retry option.

**"How does frustration detection work technically?"**
> The frontend's `useTelemetry` hook instruments the editor with event listeners for every keystroke, paste, and code run. A React `useEffect` with `setInterval` tracks idle time in real time. These four signals — idle seconds, backspace rate, paste count, run count — are combined in a weighted formula to produce a 0–1 frustration score. This score is sent with every AI request, and the backend's affect routing function uses it to choose between gentle, Socratic, and challenge modes.

**"Why Groq instead of OpenAI?"**
> Speed. Groq's LPU inference hardware runs LLaMA-class models 5–10× faster than GPU-based providers. For a tutoring system where response latency directly affects student experience, this matters. Kimi K2 is also significantly stronger at code reasoning than GPT-4o-mini at a comparable price point.

**"What's LangGraph doing here that a simple function couldn't?"**
> Two things: (1) MemorySaver — persistent conversation state keyed by thread_id means the AI accumulates context across API calls without a database query per turn. (2) The StateGraph pattern makes the pipeline explicit and testable — each node has a clear input/output contract. Adding a new mode (e.g., `generate_lesson`) is a matter of adding one node and one edge — no refactoring of existing code.

**"Isn't this just a ChatGPT wrapper with a fancy UI?"**
> No. ChatGPT receives text and returns text. SapioCode's AI backend runs a 4-stage pipeline before any LLM call: (1) tree-sitter parses the code into an AST, (2) a pattern detector identifies recursion, loops, missing base cases, (3) a frustration router selects a teaching tone, (4) a system prompt is assembled with line-numbered code, AST issues, and Socratic seeds. The LLM generates a response, and then a critic loop verifies it contains no code. You can't do any of this by pasting code into ChatGPT.

**"What happens if two students submit identical code?"**
> The viva questions will be similar (since they're generated from the same AST), but the oral defense still works — each student must independently explain the code structure, variable purposes, and edge cases. A student who copied can't explain "why did you use a temporary variable on line 2?" if they don't understand the swap pattern. The LLM evaluation is semantic, not keyword-based.

**"How would this scale to 500 students?"**
> The bottleneck is LLM inference, not our backend. Each AI call is stateless (LangGraph state is in memory per worker). The frontend is on Vercel's CDN (globally distributed). The auth backend uses MongoDB Atlas (horizontally scalable). For LLM scaling: Groq already handles thousands of concurrent requests. The dual-model failover means we can distribute load across models. For 500 concurrent students, you'd add Render workers and potentially a Redis-backed MemorySaver instead of in-process memory.

**"What if a student gives a vague answer like 'it does the thing'?"**
> The LLM evaluates against AST-derived ground truth. It's not looking for keywords — it's looking for semantic understanding of specific concepts. "It does the thing" would receive a FAIL score because it doesn't demonstrate understanding of the function's purpose, the variable roles, or the algorithm choice. The feedback screen would show zero matched concepts and list all the concepts the student missed.
